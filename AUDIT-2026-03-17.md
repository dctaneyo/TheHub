## 2. CRITICAL — Security Issues

### 2.1 `getJwtSecret()` duplicated across files
The JWT secret getter is defined identically in **two places**:
- `src/lib/auth.ts:4-11`
- `src/lib/socket-server.ts:28-35`

**Risk:** If one is changed and the other isn't, tokens signed by one module can't be verified by the other.  
**Fix:** Export from `auth.ts` only. Import in `socket-server.ts`.

### 2.2 Silent `catch {}` blocks — 169 occurrences across 52 files
Empty catch blocks swallow errors silently. Critical in:
- `src/lib/socket-server.ts:262` — heartbeat DB write failure silently ignored
- `src/lib/socket-server.ts:319` — disconnect session update failure silently ignored
- Multiple API routes in `src/app/api/data-management/` — data operations that fail silently

**Fix:** At minimum, add `console.error` in catch blocks for server-side code. Consider Sentry capture for critical paths.

### 2.3 CSP header allows `'unsafe-inline'` and `'unsafe-eval'`
In `next.config.ts:22`:
```
script-src 'self' 'unsafe-inline' 'unsafe-eval'
```
This significantly weakens XSS protection. `'unsafe-eval'` is especially dangerous.  
**Fix:** Use nonces for inline scripts if needed, or remove `'unsafe-eval'` (test that nothing breaks).

### 2.4 Data Management API routes lack permission checks
Many routes in `src/app/api/data-management/` use `getSession()` (basic auth check) but NOT `getAuthSession()` + `requirePermission()`. Routes like `drop-tables`, `purge-conversations`, `clear-sessions` are destructive admin operations that should verify the `DATA_MANAGEMENT_ACCESS` permission.

### 2.5 No session rotation on privilege change
When an ARL's role or permissions are changed, existing JWT tokens still carry the old `role` claim until they expire (24h). The `requirePermission()` helper reads fresh from DB which is good, but the middleware route-guard at `middleware.ts:168` still uses the stale JWT `userType`.

## 3. BUGS & Discrepancies

### 3.1 Dual `NotificationType` definitions — naming mismatch
Two separate notification type enums exist with **different naming conventions**:
- `src/lib/notifications.ts` — uses `snake_case`: `"task_due_soon"`, `"new_message"`, etc.
- `src/lib/notification-helpers.ts` — uses `camelCase`: `"taskDueSoon"`, `"newMessage"`, etc.

The `notification-helpers.ts` file is **never imported anywhere** — it's completely dead code. But this mismatch would cause bugs if anyone tried to use it.

### 3.2 `notification-helpers.ts` is entirely dead code
`shouldSendNotification()` and `getUserNotificationTypes()` are exported but **never imported** by any file. Notification creation in `notifications.ts` doesn't check user preferences at all — notifications are always created regardless of user settings.

**Impact:** The notification preferences UI (toggles for task_due_soon, etc.) saves to the DB but the backend **never reads them** when creating notifications. Users can toggle preferences but they have no effect.

### 3.3 `loading-messages.ts` and `pagination.ts` are dead code
- `src/lib/loading-messages.ts` — not imported anywhere
- `src/lib/pagination.ts` — not imported anywhere

### 3.4 API response format inconsistency
A standardized response helper exists at `src/lib/api-response.ts` (`apiSuccess` / `apiError` with `{ ok, data }` shape) but it's only used in **4 out of ~90 API routes**. The other ~86 routes use raw `NextResponse.json({ error: ... })`. This means the frontend has to handle two different response shapes.

### 3.5 Auth helper inconsistency across API routes
API routes inconsistently use two different auth functions:
- **45 routes** use `getSession()` (from `@/lib/auth`) — no tenant context
- **39 routes** use `getAuthSession()` (from `@/lib/api-helpers`) — includes tenant context

Routes using `getSession()` skip tenant-scoping, which could be a security issue in multi-tenant mode.

### 3.6 `createNotificationBulk()` doesn't broadcast via WebSocket
`createNotification()` calls `broadcastNotification()` after DB insert, but `createNotificationBulk()` (used for location_online/offline events) only inserts to DB — no WebSocket broadcast. Bulk notifications won't appear in real-time in the notification bell.

### 3.7 `getNotifications()` builds query twice
In `src/lib/notifications.ts:130-158`, the query is built once (lines 130-136) then entirely rebuilt if conditions > 1 (lines 150-157). The first query is always thrown away since `conditions` always has at least 1 item (the userId).

## 5. Architecture & Code Quality

### 5.1 Monolithic meeting component — 86KB / 1,927 lines
`src/components/meeting-room-livekit-custom.tsx` is a single 86KB file. This should be broken into sub-components:
- Pre-join lobby
- Video grid / gallery view
- Chat panel
- Q&A panel
- Controls bar
- Participant list
- Settings modal

### 5.2 Monolithic ARL page
`src/app/arl/page.tsx` manages ~20+ views via a single state variable. This is a maintenance burden. Consider route-based navigation or at minimum a view-router pattern.

### 5.3 Large component files throughout
Multiple components exceed 20KB:
- `restaurant-chat.tsx` (47KB)
- `remote-viewer.tsx` (33KB)
- `user-management.tsx` (31KB)
- `gamification-hub.tsx` (30KB)
- `analytics-dashboard.tsx` (30KB)
- `remote-login.tsx` (29KB)
- `messaging.tsx` (27KB)
- `data-management.tsx` (26KB)

### 5.4 No database connection pooling / WAL checkpoint management
SQLite with WAL mode is good, but there's no `PRAGMA wal_checkpoint` call. For long-running Railway deployments, the WAL file can grow unbounded.

### 5.5 Migrations use `try/catch {}` pattern extensively
`src/lib/db/index.ts` wraps ADD COLUMN calls in empty catch blocks (21 occurrences). This is intentional (idempotent migrations) but means **any** migration SQL error is silently swallowed, including typos in column names.

## 6. Performance Concerns

### 6.1 Synchronous DB calls in Socket.io connection handler
`src/lib/socket-server.ts:149` — on every location connect, does a synchronous `db.select().from(schema.arls)...all()` to get ALL active ARLs for notification creation. This blocks the event loop during high-connection-rate scenarios.

### 6.2 `getNotificationCounts()` — 3 separate DB queries
`src/lib/notifications.ts:167-193` makes three separate COUNT queries. This could be a single query:
```sql
SELECT COUNT(*) as total, SUM(CASE WHEN is_read=0 THEN 1 ELSE 0 END) as unread, 
       SUM(CASE WHEN is_read=0 AND priority='urgent' THEN 1 ELSE 0 END) as urgent 
FROM notifications WHERE user_id = ?
```

### 6.3 N+1 in `createNotificationBulk`
Called with all ARL IDs on every location connect/disconnect. For 50 ARLs, this inserts 50 rows one-by-one. Drizzle's `.values()` should batch these, but verify.

### 6.4 No request timeout on custom server
`server.ts` creates a bare `http.createServer()` with no `server.timeout` or `server.keepAliveTimeout` set. Long-running requests can hold connections indefinitely.

## 8. Feature Opportunities — Fun & Exciting

### 8.1 🎯 Actually Wire Up Notification Preferences (Bug → Feature)
The UI already exists to configure notification preferences. The DB schema stores them. The helper functions are written. The only missing piece is calling `shouldSendNotification()` before creating notifications. Wiring this up would be a high-impact, low-effort win.

### 8.2 📊 Real-time Analytics Dashboard
The schema has `meetingAnalytics`, `broadcastViewers`, `dailyLeaderboard`, etc. A live analytics dashboard showing:
- Active locations right now (already tracked via sessions)
- Task completion velocity (real-time as tasks are checked off)
- Message volume (trending conversations)
- Meeting attendance patterns

### 8.3 🏆 Achievement System (Partially Built)
API route exists at `/api/achievements/route.ts` and `/api/gamification/route.ts`. The frontend has `celebrations.tsx` and `gamification-hub.tsx`. Could expand with:
- **Milestone badges** — "First 100% day", "Week streak", "Speed demon" (completed before due time)
- **Team achievements** — All locations hit 100% on same day
- **Seasonal challenges** — Monthly goals with special rewards

### 8.4 📱 Push Notifications for Locations (Currently ARL-Only)
`pushSubscriptions` schema only stores ARL user IDs. Extending to location kiosks would enable:
- Task due reminders on the kiosk (even when app is in background/screensaver)
- Emergency broadcast alerts that can't be missed
- New message notifications for the restaurant

### 8.5 📅 Meeting Scheduling + Calendar Integration
`scheduled_meetings` table exists in migrations (migration 020) but there's no corresponding schema export. The `scheduled-meetings.tsx` component exists. Completing this would enable:
- Schedule future meetings with invites
- Calendar view of upcoming meetings
- Recurring meeting support (weekly standup, etc.)
- Meeting reminders via notification system

### 8.6 🌍 Full Multi-Tenant Onboarding
The multi-tenant infrastructure is largely built (tenant table, middleware routing, subdomain extraction). Missing pieces:
- Tenant signup/onboarding flow
- Tenant admin panel (admin subdomain routes exist but are stubs)
- Per-tenant branding (primaryColor, logoUrl, appTitle in schema but not applied in UI)
- Tenant billing/plan enforcement

### 8.7 🔍 Smart Search with Filters
`global-search.tsx` exists but could be enhanced:
- Search across messages, tasks, forms, and locations in one unified search
- Filter by date range, type, sender
- Recent searches history
- Keyboard shortcut (Cmd+K / Ctrl+K)

### 8.8 📈 Weekly Digest Email
Schema has `weeklyReport` preference and `emailNotifications` toggle. The report generator (`report-generator.ts`) is built. Connecting these for a weekly email summary would be valuable:
- Top performing locations
- Tasks completed vs missed
- Leaderboard standings
- Upcoming meetings

### 8.9 🎨 Dark Mode Toggle Enhancement
`theme-toggle.tsx` exists. The seasonal themes (`seasonal-theme.tsx`) are creative. Could add:
- Per-location theme preferences (saved to DB)
- Custom brand-color themes based on tenant `primaryColor`
- High-contrast / accessibility mode

### 8.10 🔄 Offline-First Task Completion
The offline sync infrastructure (`offline-queue.ts`, `use-offline-sync.ts`) is well-built but could be enhanced:
- Optimistic UI updates (show task as complete immediately, sync in background)
- Conflict resolution (what if ARL deletes task while location completes it offline)
- Visual indicator of pending sync items
- Background sync via service worker (not just on-focus)

## 10. Summary

**The Hub is an impressively feature-rich platform** for franchise management with real-time communication, task management, video meetings, gamification, and multi-tenant support. The architecture is sound — good use of SQLite + WAL for single-server deployments, Socket.io for real-time features, and a clean component structure.

**Top 3 priorities:**
1. **Security** — Consolidate JWT secret, add permission checks to data-management routes, tighten CSP
2. **Wire up notification preferences** — The UI/DB/helpers all exist but aren't connected
3. **Test coverage** — The codebase is large enough that automated tests would prevent regressions

**Biggest opportunity:** The meeting scheduling and multi-tenant onboarding features are 70-80% built. Finishing them would significantly expand the product's value.
